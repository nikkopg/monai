<objective>
Patch the flat-commands manifest-resolution bug (open-gsd/gsd-core#1858). This project
installed GSD Core with Claude Code's flat command layout — `.claude/commands/gsd-<stem>.md`
(no nested `gsd/` subfolder) — which `_resolveManifest()` in
`.claude/gsd-core/bin/lib/capability-state.cjs` does not detect. Neither the nested source
tree (`<repoRoot>/commands/gsd/*.md`) nor the installed-runtime skills dir
(`<configDir>/skills/gsd-*/SKILL.md`) exists here, so `_resolveManifest` returns an EMPTY
manifest. That collapses the full profile's surfaced skill set to empty, so
`resolveCapabilityState` reports `surfaced:false` / `active:false` for every capability with
skill dependencies — even when `.planning/config.json` has `graphify.enabled: true`.

Fix: add a THIRD, purely-additive fallback branch to `_resolveManifest` that scans the parent
commands directory for flat `gsd-<stem>.md` files and builds a manifest in the SAME shape the
other two paths already emit. This is the upstream issue's own proposed fix (not yet merged).

Purpose: Restore capability surfacing (graphify + 7 others) under the flat command layout.
Output: Patched `_resolveManifest` with a flat-commands fallback; `graphify status` returns
real status instead of `{"disabled": true}`.
</objective>

  STEP A — export the existing parser (install-profiles.cjs):

  the "Shared internals" comment. The function is already defined (~line 98); this only exposes it.
  Do not modify the function body.

  STEP B — import it (capability-state.cjs):
  Extend the existing destructure from `installProfilesMod` (currently
  `{ readActiveProfile, loadSkillsManifest, resolveProfile, parseRequires }`) to also pull in
  `parseCallsAgents`.

  STEP C — add helper `_loadFlatCommandsManifest(commandsDir)` (capability-state.cjs):
  Mirror the structure and defensive discipline of the sibling `_loadInstalledSkillsManifest`,
  but for a FLAT commands directory of `gsd-<stem>.md` files (not `gsd-<stem>/SKILL.md` subdirs):
  - Start with `const manifest = new Map()`. If `commandsDir` does not exist (fs.existsSync), or
    `fs.readdirSync(commandsDir, { withFileTypes: true })` throws, return the empty manifest.
  - For each entry: skip unless `entry.isFile()` AND `entry.name` starts with `gsd-` AND ends
    with `.md`. (Non-file, non-gsd, and non-md entries are skipped so user files are never pulled in.)
  - Stem extraction: `entry.name.slice(4, -3)` strips the leading `gsd-` (4 chars) and trailing
    `.md` (3 chars). Skip the entry when the stem is empty.
  - Read the file with fs.readFileSync utf8. On success:
    `manifest.set(stem, parseRequires(content))` and
    `manifest.set(\`_calls_agents_${stem}\`, parseCallsAgents(content))`.
    This matches `loadSkillsManifest` exactly (the flat files ARE the real command bodies, so
    agent refs are recoverable — unlike installed SKILL.md, which is why the installed path emits []).
  - On read failure (catch): `manifest.set(stem, [])` and `manifest.set(\`_calls_agents_${stem}\`, [])`
    — parity with `loadSkillsManifest`'s catch branch.
  - Return manifest.
  Place this helper immediately after `_loadInstalledSkillsManifest`.

  STEP D — wire the third branch into `_resolveManifest(commandsGsdDir, configDir)`:
  Keep the first branch unchanged: if `fs.existsSync(commandsGsdDir)` return
  `loadSkillsManifest(commandsGsdDir)`. Then compute `const installed = _loadInstalledSkillsManifest(configDir)`
  and, ONLY when `installed.size > 0`, return it. Otherwise (both existing layouts empty) return
  `_loadFlatCommandsManifest(node_path_1.default.dirname(commandsGsdDir))`. The `installed.size > 0`
  guard is what makes the change purely additive — the flat fallback is unreachable whenever the
  installed skills dir yields entries. Update the `_resolveManifest` doc comment's resolution-order
  list to add the third (flat commands / #1858) branch.

  STEP E — export helper for parity (capability-state.cjs):
  Add `_loadFlatCommandsManifest` to `module.exports` next to the existing test-exported
  `_loadInstalledSkillsManifest`.

  Do NOT alter `_resolveCommandsGsdDir`, `resolveCapabilityState`, `resolveSurface`, or any
  activation logic. Reuse `parseRequires`/`parseCallsAgents` — do not reimplement parsing.
  </action>
