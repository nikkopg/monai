# Quick 260711-rb2 — Multi-platform holdings (platform required)

- Current data is clean: all 11 holdings have a platform, portfolio_events has 3 rows / 2 tickers — trivial backfill.

- `PortfolioEventCreate` already carries `platform_id: int | None` (from quick 260711-k35) — this task makes it required.

## Task 1 — Migration (identity change) + models

**Files:** new `alembic/versions/006_multi_platform_holdings.py`, `backend/models.py`

Migration `upgrade()` (order matters):
1. `op.alter_column("holdings", "platform_id", nullable=False)` — all rows already have a platform. (If any NULL existed the migration would fail loudly; that's acceptable — data is clean.)
2. Drop the global ticker uniqueness: `op.drop_constraint("holdings_ticker_key", "holdings", type_="unique")` and `op.drop_index("ix_holdings_ticker", table_name="holdings")`.
3. Recreate a **non-unique** ticker index (price refresh + lookups still filter by ticker): `op.create_index("ix_holdings_ticker", "holdings", ["ticker"])`.
4. Composite uniqueness: `op.create_unique_constraint("uq_holdings_ticker_platform", "holdings", ["ticker", "platform_id"])`.
5. `op.add_column("portfolio_events", sa.Column("platform_id", sa.Integer(), nullable=True))`.
6. Backfill from the (currently one-per-ticker) holding:
   `op.execute("UPDATE portfolio_events e SET platform_id = h.platform_id FROM holdings h WHERE h.ticker = e.ticker")`.
7. `op.alter_column("portfolio_events", "platform_id", nullable=False)`, add FK
   `op.create_foreign_key("fk_portfolio_events_platform", "portfolio_events", "platforms", ["platform_id"], ["id"])`,
   and `op.create_index("ix_portfolio_events_platform_id", "portfolio_events", ["platform_id"])`.

`downgrade()` reverses in strict reverse order (drop event FK/index/column; drop composite unique; drop non-unique ticker index; restore `unique=True` ticker constraint+index; `platform_id` nullable again).
> NOTE in the migration docstring: downgrade only succeeds if no duplicate-ticker rows exist yet (restoring `unique(ticker)` would fail otherwise) — inherent, tested immediately after upgrade before any duplicate is added.

`backend/models.py`:
- `Holding`: remove `unique=True` from the `ticker` column (keep `index=True`); set `platform_id` `nullable=False`; add
  `__table_args__ = (UniqueConstraint("ticker", "platform_id", name="uq_holdings_ticker_platform"),)`.
- `PortfolioEvent`: add `platform_id: Mapped[int] = mapped_column(ForeignKey("platforms.id"), nullable=False, index=True)`.

**Verify (rebuilt image):** `alembic upgrade head`; `\d holdings` shows composite unique + platform_id NOT NULL; `\d portfolio_events` shows platform_id NOT NULL; then `downgrade -1` and `upgrade head` clean.

## Task 2 — Ledger: recompute per position + event write (TDD)

**Files:** `backend/portfolio.py`, `backend/writes.py`, `backend/tests/test_portfolio.py`, `backend/tests/test_write_tools.py`

- `recompute_holding_from_events(db, ticker, platform_id)` — new required `platform_id` param:
  - filter events: `.where(PortfolioEvent.ticker == ticker, PortfolioEvent.platform_id == platform_id)`.
  - upsert the holding by `(ticker, platform_id)`; when creating, set `platform_id=platform_id` (it's NOT NULL now).
  - keep the zero-qty-retention (D-04) and avg-cost (D-02) logic unchanged.
- `apply_add_portfolio_event` (`backend/writes.py`):
  - the `PortfolioEvent(...)` constructor gains `platform_id=after["platform_id"]` (required — the schema guarantees it).
  - call `recompute_holding_from_events(db, after["ticker"], after["platform_id"])`.
  - the k35 post-recompute set-block: **drop the `platform_id` part** (platform is now identity, set by recompute); **keep the `asset_type` set-when-provided** part (asset_type is a plain attribute) — still needs the `db.flush()` before the lookup, and the lookup now filters by BOTH ticker and platform_id.
- `apply_delete_platform` (`backend/writes.py`): when `reassign_to` is provided it currently UPDATEs `holdings.platform_id`. It MUST also reassign events, and must not create a collision:
  - **Collision guard first** (ponytail: block, don't auto-merge): if any ticker exists on BOTH the deleted platform and `reassign_to`, `raise ValueError("Cannot reassign: {ticker} already exists on the target platform — merge positions manually first.")`. (`# ponytail: reject colliding reassignment; position-merge is a later feature.`)
  - then `UPDATE holdings SET platform_id=... WHERE platform_id=...` AND `UPDATE portfolio_events SET platform_id=... WHERE platform_id=...` (keep the single audit row; include reassigned_count).

**Tests:**
- `test_portfolio.py`: two platforms, same ticker, independent buys → two holdings with independent qty/avg_cost; recompute of one doesn't touch the other.
- `test_write_tools.py`: `apply_add_portfolio_event` with platform_id creates the position on that platform; a second event same ticker different platform creates a second position; `apply_delete_platform` colliding reassignment raises ValueError.
