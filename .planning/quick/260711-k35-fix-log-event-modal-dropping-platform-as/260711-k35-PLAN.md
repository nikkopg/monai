<objective>
Fix the Log-event modal (HoldingModal) silently dropping Asset type and Platform so a buy
event actually sets the created holding's platform_id and asset_type — restoring the
"group holdings by platform" feature from its primary entry point.

Root cause (already diagnosed in Phase 5 browser UAT, do NOT re-investigate):
- Frontend `handleSubmit` (HoldingModal.tsx:61) builds the POST body from only
  {ticker, event_type, quantity, price, date} — assetType/platformId/notes are dropped.
- `PortfolioEventCreate` (schemas.py:123) has no platform_id/asset_type/notes fields.
- `apply_add_portfolio_event` (writes.py:193) calls `recompute_holding_from_events`, which
  creates the holding with only ticker/quantity/avg_cost/currency (portfolio.py:87) —
  platform_id and asset_type are never set, so every buy yields an Unassigned/null holding.

Notes decision (based on actual model): `PortfolioEvent` (models.py:168) has NO notes column,
and persisting notes would require a new migration — out of scope for a quick fix. Therefore
Notes is DROPPED from scope and the dead Notes input is removed from the modal.

Purpose: A buy logged with a platform lands in the right platform group; asset_type is set.
Output: Wired platform_id + asset_type through frontend → schema → writes; extended test;
Notes input removed.
</objective>
