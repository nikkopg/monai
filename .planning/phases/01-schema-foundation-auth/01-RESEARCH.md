## Summary

Phase 1 is pure infrastructure: no new user-visible features, but it closes three hard blockers before any write endpoint or new table can be trusted. The project already has a healthy PostgreSQL database (`accounts`, `transactions`, `date_helpers` view) with 5,609 live transaction rows and 3 accounts — **confirmed by live DB inspection**. No `alembic_version` table exists yet. All schema bootstrapping today happens via `Base.metadata.create_all()` in `db.py:33` plus a `CREATE OR REPLACE VIEW` in `init_db()`.

The central challenge is introducing Alembic non-destructively. The standard "stamp" recipe applies directly: generate a baseline migration that matches the current ORM models exactly, run `alembic stamp <rev>` on the live DB (marks it applied without touching data), then add a second migration for the five new tables. The `date_helpers` view moves from `init_db()` into that second migration via `op.execute()`. Once Alembic owns the schema, `Base.metadata.create_all()` and the view-creation code are removed from `db.py`.

Auth is a thin FastAPI dependency (`require_api_key`) added to `POST /transactions` and `POST /import`. It reads `MONAI_API_KEY` from env and uses `hmac.compare_digest` for constant-time comparison. The Next.js proxy rewrites in `next.config.js` are replaced with a catch-all App Router route handler that injects the header server-side (key stays out of the browser). The Decimal fix is a single annotated type alias that replaces `float` in `TransactionCreate.amount` and `TransactionOut.amount`, with `PlainSerializer(float, when_used='json')` so JSON output is numbers not strings.

**Primary recommendation:** Follow the two-migration + stamp pattern exactly as in D-01. Do not use `alembic revision --autogenerate` for the baseline without verifying the generated DDL matches the live schema — autogenerate can miss indexes or produce extra statements. Write the baseline migration by hand to exactly mirror what `\d accounts` and `\d transactions` show, then let migration 2 be autogenerated from the new ORM models added to `models.py`.

### Pattern 1: Alembic Introduction on Existing DB (Two-Migration + Stamp)

**What:** Generate a baseline migration representing the current schema, stamp the live DB to mark it applied without running it, then add new tables in a second migration.

**When to use:** Any time Alembic is introduced to a database that already has live data created outside Alembic.

**Step-by-step:**
```bash
# 1. Install alembic
pip install alembic

# 2. Initialize alembic layout
alembic init alembic

# 3. Edit alembic/env.py — point at DATABASE_URL + Base.metadata (see Pattern 2 below)

# 4. Generate baseline revision (autogenerate from current ORM models)
alembic revision --autogenerate -m "baseline"
# NOTE: Review the generated script carefully. It should match the live DB exactly.
# If autogenerate produces extra/wrong DDL, edit by hand to match \d accounts / \d transactions output.

# 5. STAMP the live database — marks migration as applied WITHOUT running any SQL
#    This is the key safety step: live data is untouched.
alembic stamp <revision_id_from_step_4>

# 6. Verify stamp worked
alembic current  # should show: <rev_id> (head)

# 7. Generate the second migration for new tables
alembic revision --autogenerate -m "add_new_tables"
# Add 5 new tables to models.py first, then run autogenerate

# 8. Apply the second migration on the live DB
alembic upgrade head
# Only migration 002 runs. accounts/transactions data untouched.
```

**Source:** [CITED: alembic.sqlalchemy.org/en/latest/cookbook.html] — "Working with an existing database" section; [CITED: alembic.sqlalchemy.org/en/latest/api/commands.html] — `stamp` command

**Critical warning:** Autogenerate does NOT detect all schema objects. It will NOT detect the existing `date_helpers` view (views are invisible to autogenerate). The baseline migration must include the view only in the sense that it should NOT attempt to `CREATE VIEW date_helpers` — the view already exists. Migration 002 should create the new tables AND contain the `date_helpers` view definition via `op.execute()` so that a fresh database gets the view too. This means migration 002 must use `op.execute("CREATE OR REPLACE VIEW date_helpers AS ...")`.

### Pattern 2: Alembic env.py for DATABASE_URL + psycopg3

**What:** Configure `env.py` to read `DATABASE_URL` from environment rather than hardcoding in `alembic.ini`.

```python
# alembic/env.py — sync (not async — project uses sync SQLAlchemy engine)
import os
from logging.config import fileConfig

from sqlalchemy import engine_from_config, pool
from alembic import context

# Import the project's Base so autogenerate sees all models
from backend.models import Base

config = context.config
fileConfig(config.config_file_name)

target_metadata = Base.metadata


def run_migrations_online():
    # Read DATABASE_URL from environment; override alembic.ini
    db_url = os.environ.get(
        "DATABASE_URL",
        "postgresql+psycopg://monai:monai@localhost:5434/monai",
    )
    # configparser treats % as interpolation — escape if present in password
    configuration = config.get_section(config.config_ini_section)
    configuration["sqlalchemy.url"] = db_url.replace("%", "%%")

    connectable = engine_from_config(
        configuration,
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    with connectable.connect() as connection:
        context.configure(connection=connection, target_metadata=target_metadata)
        with context.begin_transaction():
            context.run_migrations()
```

**Key detail:** The project uses `postgresql+psycopg://` (psycopg3 sync driver). This is compatible with `engine_from_config` — no async env.py needed. [ASSUMED: psycopg3 sync driver works identically to psycopg2 for Alembic's purposes; the connection string prefix difference is the only change.]

**Source:** [CITED: alembic.sqlalchemy.org/en/latest/api/config.html]; [CITED: github.com/sqlalchemy/alembic/discussions/1149]

### Pattern 3: FastAPI `require_api_key` Dependency

**What:** A FastAPI dependency that extracts `MONAI_API_KEY` from the request header and rejects with 401 if missing or wrong.

```python
# backend/auth.py
import hmac
import os
from fastapi import Header, HTTPException, Security
from fastapi.security import APIKeyHeader

_API_KEY_HEADER = APIKeyHeader(name="MONAI_API_KEY", auto_error=False)

_CONFIGURED_KEY: str = os.environ.get("MONAI_API_KEY", "")


def require_api_key(api_key: str | None = Security(_API_KEY_HEADER)) -> None:
    """FastAPI dependency — call on write routes only."""
    if not _CONFIGURED_KEY:
        raise RuntimeError("MONAI_API_KEY env var is not set — refusing all writes.")
    if not api_key or not hmac.compare_digest(api_key, _CONFIGURED_KEY):
        raise HTTPException(status_code=401, detail="Invalid or missing API key")
```

**Usage in main.py:**
```python
from backend.auth import require_api_key
from fastapi import Depends

@app.post("/transactions", response_model=TransactionOut, status_code=201,
          dependencies=[Depends(require_api_key)])
def create_transaction(...): ...

@app.post("/import", response_model=ImportResponse,
          dependencies=[Depends(require_api_key)])
async def import_csv(...): ...
# POST /query does NOT get the dependency (D-06)
```

**Source:** [CITED: fastapi.tiangolo.com/reference/security/] — `APIKeyHeader`; [CITED: bugs.python.org/issue40791] — `hmac.compare_digest` constant-time

**Why `dependencies=[Depends(...)]` not `Depends` in signature:** Using the `dependencies=` parameter keeps the function signature clean and doesn't inject the return value into the handler body. Both approaches work; this is more readable for auth.

**Why `auto_error=False`:** Lets us return a 401 with a consistent message rather than FastAPI's default 403. Without `auto_error=False`, a missing header returns 403 Forbidden (wrong status for API key auth).

### Pattern 4: Shared Decimal Money Type Alias (Pydantic v2)

**What:** A single `Annotated[Decimal, PlainSerializer(...)]` alias that serializes as a JSON number (not string). Reused across all schemas.

**Why this is needed:** Pydantic v2 serializes `Decimal` fields as **strings** in JSON by default (confirmed as intentional behavior per [CITED: github.com/pydantic/pydantic/issues/7457]). The frontend expects numbers. The fix is a `PlainSerializer` that returns `float` when serializing to JSON.

**Precision risk:** Converting `Decimal("123456789.99")` to `float` for JSON is safe at 2dp IDR amounts — `float64` has 15-17 significant digits, IDR amounts in the dataset are well within that range. For `Numeric(28,8)` crypto quantities, there is theoretical precision loss beyond 15 sig figs, but this is accepted (D-14).

```python
# backend/schemas.py — add near the top
from decimal import Decimal
from typing import Annotated
from pydantic import PlainSerializer

# Serializes Decimal as a JSON number (float), not string.
# Used for all money and quantity fields across schemas.
MoneyDecimal = Annotated[
    Decimal,
    PlainSerializer(lambda x: float(x), return_type=float, when_used="json"),
]

# Then use in models:
class TransactionCreate(BaseModel):
    amount: MoneyDecimal = Field(...)  # was: float
    ...

class TransactionOut(BaseModel):
    amount: MoneyDecimal  # was: float
    ...
```

**Source:** [CITED: github.com/pydantic/pydantic/discussions/8505] — Decimal precision + PlainSerializer pattern; [CITED: docs.pydantic.dev/2.11/api/standard_library_types/] — Decimal type behavior

### Pattern 5: Next.js App Router Catch-All Proxy with Header Injection

**What:** Replace (or supplement) `next.config.js` rewrites with an App Router route handler that forwards all `/api/*` calls to the FastAPI backend while injecting the `MONAI_API_KEY` header server-side.

**Why the rewrite proxy isn't enough:** Next.js `rewrites()` in `next.config.js` are pure path rewrites — they cannot add request headers. A route handler runs server-side Node.js and has access to `process.env`.

```typescript
// ui/app/api/[...proxy]/route.ts
import { NextRequest, NextResponse } from "next/server";

const BACKEND = process.env.MONAI_API || "http://127.0.0.1:8001";
const API_KEY = process.env.MONAI_API_KEY || "";

async function forwardRequest(req: NextRequest, segments: string[]): Promise<NextResponse> {
  const path = segments.join("/");
  const url = new URL(req.url);
  const target = `${BACKEND}/${path}${url.search}`;

  const headers = new Headers(req.headers);
  headers.set("MONAI_API_KEY", API_KEY);
  // Remove Next.js internal headers that confuse upstream
  headers.delete("host");

  const init: RequestInit = { method: req.method, headers };
  if (req.method !== "GET" && req.method !== "HEAD") {
    init.body = await req.arrayBuffer(); // Must read body explicitly for POST/PUT
  }

  const upstream = await fetch(target, init);
  return new NextResponse(upstream.body, {
    status: upstream.status,
    headers: upstream.headers,
  });
}

export async function GET(req: NextRequest, { params }: { params: { proxy: string[] } }) {
  return forwardRequest(req, params.proxy);
}
export async function POST(req: NextRequest, { params }: { params: { proxy: string[] } }) {
  return forwardRequest(req, params.proxy);
}
// Add DELETE, PATCH, PUT as needed
```

**docker-compose.yml change needed:** Add `MONAI_API_KEY` to the `frontend` service environment block.

**Source:** [CITED: nextjs.org/docs/app/getting-started/route-handlers]; [CITED: github.com/vercel/next.js/discussions/51697] — catch-all proxy in App Router


**docker-compose.yml** already has `depends_on: db: condition: service_healthy` — this ensures PostgreSQL is ready before the backend starts, which means `alembic upgrade head` will not fail on connection.

**Alternative (manual):** Keep current Dockerfile `CMD` and document `docker exec monai-backend alembic upgrade head` in the runbook. Simpler but requires manual step.

**Recommendation:** Use the entrypoint approach — it makes `docker compose up` idempotent and safe for repeat deployments. [ASSUMED: entrypoint approach is the community standard for Alembic + Docker; verified against multiple examples.]

### Anti-Patterns to Avoid

- **Running `alembic upgrade head` without stamping first:** Alembic will see a clean `alembic_version` table and try to run migration 001 (baseline), which will fail with "table already exists". Always stamp the existing DB before first upgrade.
- **Using `alembic autogenerate` for the baseline without verifying output:** Autogenerate will not detect the `date_helpers` view and may generate spurious differences (e.g., index name differences). Always `\d` inspect the live schema and compare.
- **Using `secrets.compare_digest` instead of `hmac.compare_digest`:** Both are constant-time in Python 3.x, but `hmac.compare_digest` has been available since Python 3.3 and is the documented standard for this use case.
- **Putting `MONAI_API_KEY` in `next.config.js` `rewrites()`:** Rewrites cannot add headers. Only a route handler (server-side) or Next.js middleware can inject headers.
- **Keeping `Base.metadata.create_all()` in `db.py` after Alembic is introduced:** If both run, `create_all` will silently succeed (tables exist), but any future schema drift between ORM models and migrations will be masked.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Database migration versioning | Custom SQL migration scripts with version table | `alembic` | Handles dependency graph, autogenerate, stamp, history — battle-tested at scale |
| Constant-time string comparison | `api_key == stored_key` | `hmac.compare_digest` | Python built-in; `==` leaks timing information via early exit |
| Decimal JSON serialization | Custom `json.dumps` encoder | `PlainSerializer` in Pydantic v2 `Annotated` type | Pydantic handles serialization context; custom encoder bypasses schema validation |
| API key header extraction | Manual `request.headers.get(...)` in each handler | `fastapi.security.APIKeyHeader` + `Security` | Integrates with OpenAPI docs; consistent 401 handling |

**Key insight:** Every one of these is a solved problem with stdlib or project-already-present libraries. Zero new complex dependencies needed beyond `alembic`.

## Common Pitfalls

### Pitfall 1: Baseline Migration Fails on Existing Tables

**What goes wrong:** Running `alembic upgrade head` without stamping first causes migration 001 to execute its `CREATE TABLE accounts` statement against the live DB, which fails with `relation "accounts" already exists`.

**Why it happens:** Alembic checks `alembic_version` table. If empty/absent, it runs all migrations from the beginning.

**How to avoid:** Always run `alembic stamp <baseline_rev>` on the existing DB BEFORE running `alembic upgrade head` for the first time. The stamp creates the `alembic_version` table and inserts the revision ID without executing any SQL.

**Warning signs:** `alembic current` returns nothing / empty; `alembic_version` table does not exist.

### Pitfall 2: Autogenerate Detects Spurious Differences

**What goes wrong:** Autogenerate generates ALTER/DROP statements for things that are actually correct (index names differ between SQLAlchemy default and actual DB, or the view appears as a table to be dropped).

**Why it happens:** Alembic autogenerate reflects the live DB and compares with ORM metadata. The live DB was created by `create_all()` which may use slightly different naming conventions. Views are not visible to autogenerate by default but the view table-like appearance can cause false positives.

**How to avoid:** For the baseline migration, inspect `\d accounts` and `\d transactions` output (shown above in Research) and write the baseline by hand if autogenerate produces anything unexpected. Run `alembic check` after stamping to verify zero diff.

**Warning signs:** Baseline autogenerate script contains `DROP INDEX`, `ALTER TABLE`, or `CREATE VIEW` statements.

### Pitfall 3: psycopg3 Connection String in alembic.ini

**What goes wrong:** Alembic `alembic.ini` stores the URL as `postgresql+psycopg://...` but the `%` character in passwords gets interpreted as a Python configparser interpolation sequence, causing `configparser.InterpolationSyntaxError`.

**Why it happens:** `configparser` uses `%` as its interpolation delimiter; `%(foo)s` is a valid config template.

**How to avoid:** Override the URL in `env.py` via `config.set_main_option("sqlalchemy.url", url.replace("%", "%%"))`. Leave `alembic.ini` `sqlalchemy.url` as a placeholder.

**Warning signs:** `InterpolationSyntaxError` or `ValueError: invalid interpolation syntax`.

### Pitfall 4: Pydantic v2 Serializes Decimal as String

**What goes wrong:** Switching `amount: float` to `amount: Decimal` in schemas causes the JSON response to return `"amount": "123456789.00"` (string) instead of `"amount": 123456789.00` (number). Frontend chart libraries break.

**Why it happens:** Pydantic v2 intentionally serializes `Decimal` as strings in JSON mode to preserve full precision. This was changed from v1 behavior.

**How to avoid:** Use the `MoneyDecimal` alias with `PlainSerializer(lambda x: float(x), when_used="json")`. Validated in Python mode, float in JSON mode.

**Warning signs:** Frontend receiving amounts as strings; `JSON.parse` results in numbers but `typeof amount === "string"`.

### Pitfall 5: models.py Transaction.amount Type Annotation vs Column Type Mismatch

**What goes wrong:** The current `models.py` has `amount: Mapped[float] = mapped_column(Numeric(18, 2))` — the Python type hint says `float` but the DB column is `Numeric`. SQLAlchemy 2.0 uses the `Mapped[T]` annotation as the Python-side type. Changing the schema to return `Decimal` requires fixing both the annotation (`Mapped[Decimal]`) and the schema field type.

**How to avoid:** Update `models.py` `Transaction.amount` annotation from `Mapped[float]` to `Mapped[Decimal]`; import `Decimal` from `decimal` module.

**Warning signs:** mypy/pyright errors; unexpected `float` values appearing from ORM queries.

### Pitfall 6: Next.js `rewrites()` Cannot Inject Headers

**What goes wrong:** Developer adds `MONAI_API_KEY` logic to `next.config.js` rewrites — it doesn't work; the header is never sent.

**Why it happens:** `rewrites()` in `next.config.js` is a pure path rewrite at the Next.js routing layer. It has no mechanism for header injection.

**How to avoid:** Replace rewrites with App Router route handler (`ui/app/api/[...proxy]/route.ts`) that explicitly calls `headers.set("MONAI_API_KEY", process.env.MONAI_API_KEY)`.

**Warning signs:** Write endpoints receiving 401 even when the frontend is "sending the key"; network tab showing no `MONAI_API_KEY` header in browser requests.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `create_all()` for schema bootstrap | Alembic migrations with stamp for existing DB | Alembic released 2010; standard since SQLAlchemy 1.0 era | Enables safe incremental schema evolution |
| `Decimal` serialized as strings in Pydantic v2 | `PlainSerializer` annotated type alias | Pydantic v2 (2023) changed behavior from v1 | Must use `PlainSerializer` — `json_encoders` is v1 API |
| `APIKeyHeader` with direct string comparison | `APIKeyHeader` + `hmac.compare_digest` | Python 3.3+ (2012) | Constant-time is stdlib; no excuse to skip it |

**Deprecated/outdated:**
- `pydantic.ConfigDict(json_encoders=...)`: v1 pattern; does not work as expected in v2 for Decimal. Use `PlainSerializer`.
- `@app.on_event("startup")` for schema creation: Still works in FastAPI but `create_all` is the anti-pattern here; removing both together is correct.
- `next.config.js rewrites()` for header injection: Never worked for headers; must use route handler.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes — API key auth | `APIKeyHeader` + `hmac.compare_digest` |
| V3 Session Management | no — stateless single-key API | — |
| V4 Access Control | yes — read/write separation | Per-route `Depends(require_api_key)` on write routes only |
| V5 Input Validation | yes — amount, currency, dates | Pydantic v2 schema validation on `TransactionCreate` |
| V6 Cryptography | yes — constant-time comparison | `hmac.compare_digest` (stdlib) — never custom |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Timing attack on API key comparison | Information Disclosure | `hmac.compare_digest` — constant-time regardless of match length |
| API key leaking to browser via JS | Information Disclosure | Server-side Next.js route handler; key only in `process.env` |
| Missing `MONAI_API_KEY` env var (empty key accepts any request) | Elevation of Privilege | Explicit check: `if not _CONFIGURED_KEY: raise RuntimeError(...)` in `require_api_key` |
| Migration running against wrong DB (dev vs prod URL) | Tampering | `alembic upgrade head` reads `DATABASE_URL` from env — same env var used by app; explicit backup step in runbook |
