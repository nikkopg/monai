**Gathered:** 2026-06-21

<domain>
## Phase Boundary

Make the database safe to evolve and lock down writes. This phase delivers:
- **Alembic** introduced onto the *existing, populated* Postgres volume (~5,608 rows) without destroying `transactions`/`accounts` data.
- Five new tables created via migration: `audit_log`, `proposals`, `holdings`, `portfolio_events`, `price_cache`.
- **API-key auth** (`MONAI_API_KEY`) enforced on all write endpoints; reads stay open.
- **Decimal**-correct money handling end-to-end (close the float-in-transit leak at the Pydantic/JSON edge).
- `Base.metadata.create_all()` removed from `db.py` тАФ schema becomes fully Alembic-managed.

This is foundational infrastructure that unblocks Phases 2тАУ6. **In scope = HOW to build the above.** New capabilities (the agentic loop, UI, investments fetching, MCP) belong to their own phases. The five tables are *created* here but *consumed* later (proposals/audit in Phase 2; holdings/events/price_cache in Phase 5).

</domain>

<decisions>
## Implementation Decisions

### Alembic Migration Strategy
- **D-01:** Two migrations + stamp. Migration 1 = autogenerated baseline matching the **current** models (`accounts`, `transactions`); run `alembic stamp <rev>` on the live DB so the baseline is marked applied **without re-running** it. Migration 2 creates the 5 new tables. (Standard "introduce Alembic to an existing DB" recipe тАФ lowest data-loss risk.)
- **D-02:** The `date_helpers` view (currently created in `init_db()`, `db.py:14-35`) moves **into a migration** so the entire schema (tables + view) is Alembic-managed. Consistent with success criterion #4.
- **D-03:** Safety net before the first `alembic upgrade head` on real data = a **documented `pg_dump` backup step** in the runbook/README. (Pragmatic for a single-user self-hosted app; no hard requirement that every migration have a tested `downgrade()`.)
- **D-04:** `Base.metadata.create_all()` is removed from `db.py` (`db.py:33`). Schema is fully Alembic-managed after this phase (success criterion #4).

### API Key Authentication
- **D-05:** Enforcement = a **per-route FastAPI dependency** (e.g. `require_api_key`) attached to write routes (`POST`/`PUT`/`DELETE`/`PATCH`). Explicit and readable per-endpoint; reads stay open by simply not having the dependency.
- **D-06:** Public (no key) = **all GET reads** (`/health`, `/accounts`, `/transactions`) **plus `POST /query`** (read-only despite being POST). Only state-changing writes require the key. Matches success criterion #2. Today's write surface: `POST /transactions`, `POST /import`.
- **D-07:** The Next.js frontend delivers the key via a **server-side proxy that injects the `MONAI_API_KEY` header** from an env var тАФ the key never reaches the browser. Today `ui/next.config.js` uses a pure rewrite proxy; this likely needs a Next route handler / middleware to inject headers. Injecting on every proxied call is fine (harmless on public routes, required for writes).
- **D-08:** Header name is `MONAI_API_KEY` (per success criterion #2 wording). Key comparison must be constant-time. Single static key from env.

### New Table Schemas & Precision
- **D-09:** Numeric precision: **quantity `Numeric(28,8)`** (crypto-standard 8 decimals), **money `Numeric(18,2)`** (consistent with existing `transactions.amount`). Applies to holdings (avg_cost, value), prices, P&L. Note: tiny sub-rupiah per-unit token prices are not preserved at 2 dp тАФ accepted tradeoff for consistency.
- **D-10:** `audit_log` before/after and `proposals` operation payloads stored as **JSONB columns** (operations vary across transaction/account/category/holding тАФ one flexible shape beats N typed schemas).
- **D-11:** The proposal **confirm token is a separate high-entropy secret column**, not the row's UUID id. The id may appear in logs/URLs without enabling a confirm. Token is single-use + operation-scoped (carried from pre-roadmap decisions; enforced in Phase 2).
- **D-12:** A manually-set/overridden holding price lives in **`price_cache` with `source='manual'`** тАФ all prices (fetched or manual) flow through one price layer with a `source` field, giving a single read path for "current price." (Supports INV-04 fallback for reksadana / no-API instruments.)

### Decimal Conversion Boundary
- **D-13:** Retrofit reaches **all money fields now** тАФ including existing `TransactionCreate`/`TransactionOut` (`schemas.py:11,22` currently `amount: float`). Criterion #3 names "new transaction" amounts, so the create path is in scope; do it once cleanly.
- **D-14:** Decimals serialize as **JSON numbers** (e.g. `123.45`), not strings. Easiest for the JS frontend/charts; no precision loss at personal-finance magnitudes for IDR(2dp)/qty(8dp).
- **D-15:** Define **one shared annotated `Decimal` money type** (e.g. a `condecimal`/`Annotated[Decimal, ...]` alias) reused across all schemas to keep precision uniform and signal intent.

### Claude's Discretion
- Exact column lists for the 5 tables (beyond the precision/JSONB/token decisions above) тАФ derive from success criteria. Suggested skeletons: `audit_log(id, entity, entity_id, operation, before JSONB, after JSONB, created_at)`; `proposals(id UUID, token <secret>, operation, payload JSONB, status, expires_at, created_at, confirmed_at)`; `holdings(id, ticker, quantity Numeric(28,8), avg_cost Numeric(18,2), purchase_date, currency, asset_type)`; `portfolio_events(id, date, ticker, event_type, quantity Numeric(28,8), price Numeric(18,2))`; `price_cache(id, ticker, price Numeric(18,2), currency, source, fetched_at)`.
- Choice of Alembic env layout (`alembic/` location, `env.py` wiring to `DATABASE_URL` + `Base.metadata`).
- Constant-time comparison primitive (`hmac.compare_digest` or equivalent).

</decisions>

## Specific Ideas

- The five tables are forward-looking: created in Phase 1, consumed in Phase 2 (`proposals`, `audit_log`) and Phase 5 (`holdings`, `portfolio_events`, `price_cache`). Build them flexible (JSONB payloads) so Phase 2/5 don't need re-migrations for the operation taxonomy.
- `price_cache` with a `source` enum (`manual` + future `coingecko`/`yfinance`/etc.) is the seam the Phase 5 pluggable price adapters plug into.
