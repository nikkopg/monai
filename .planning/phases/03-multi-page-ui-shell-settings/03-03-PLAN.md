<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → /api/settings (proxy) | User-entered provider/keys/preferences cross to the backend via the server-side proxy |
| server (GET /api/settings) → browser | Only masked key values may reach the browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-20 | Information Disclosure | API key inputs / rendered state | high | mitigate | Key inputs are type=password; the page only ever receives masked values from GET (raw keys never round-trip per 03-02 T-03-10); the raw key the user types is sent once on Save and not persisted in client state beyond the input. |
| T-03-21 | Tampering / Availability | blank key field on Save | medium | mitigate | Save omits blank/unchanged key fields from the PUT body, so a blank field cannot clear a stored key (backend also enforces this in 03-02 T-03-12). |
| T-03-22 | Information Disclosure | MONAI_API_KEY handling | low | accept | The browser adds no auth header; the existing catch-all proxy injects MONAI_API_KEY server-side (route.ts unchanged), so the key is never exposed to client JS. |
| T-03-23 | Tampering | proxy route blast radius | low | accept | ui/app/api/[...proxy]/route.ts is explicitly not modified (RESEARCH.md Pitfall 4); /api/settings uses the existing buffered branch. |
</threat_model>
