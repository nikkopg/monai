## Spacing Scale

Standard 8-point scale only: **4, 8, 16, 24, 32, 48, 64** ‚Äî every value below is one of these, no exceptions. Where the currently shipped `ui/app/page.tsx` constants use an off-scale value, extraction into `ui/app/styles.ts` normalizes it to the nearest standard token below. This is a deliberate, minor visual change (a few px of padding/margin) made once, at extraction time.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (e.g. `‚Ä∫` step marker), checkbox right-margin, label bottom-margin |
| sm | 8px | Button row gap, input-to-button gap, nav link internal gap |
| md | 16px | Form grid gaps (`gridTemplateColumns` rows), nav-link-to-nav-link spacing |
| lg | 24px | Card padding (`card.padding`), card bottom-margin (`card.marginBottom`) ‚Äî normalized from the shipped 20px value on extraction into `ui/app/styles.ts` |
| xl | 32px | Page title bottom-margin (existing `h1` block spacing) ‚Äî normalized from the shipped 28px value on extraction |
| 2xl | 48px | Page outer padding (`main` horizontal/vertical padding) ‚Äî normalized from the shipped 40px value on extraction |

Legacy value normalization: the existing `input` and `btn` constants use off-scale padding (`input: "10px 12px"`, `btn: "10px 18px"`). On extraction into `ui/app/styles.ts` these are rounded to the nearest multiples of 4:
- `input` padding: `10px 12px` ‚Üí **`8px 12px`**
- `btn` padding: `10px 18px` ‚Üí **`8px 16px`**

There is no verbatim/unchanged carve-out for spacing ‚Äî every constant moved into `ui/app/styles.ts` must land on the standard scale. New Phase 3 elements (Nav bar, Settings page sections) must also use only the token scale above; do not introduce further off-scale values.

**Nav-specific:** horizontal nav bar height 56px (multiple of 4), 24px horizontal outer padding, 24px gap between the four nav links, sticky at top (`position: sticky; top: 0`) with the card border color as its bottom border (see Color).

## Typography

Consolidated from the sizes already in use (11/12/13/14/15/28px) down to the 4 declared here. New Phase 3 UI (Nav, Settings) must use only these ‚Äî existing components already shipped (e.g. `ProposalCard`'s 11px uppercase micro-label) are left untouched since Phase 3 does not modify them.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 400 | 1.4 |
| Body | 14px | 400 (600 for button/CTA text only) | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Weights: exactly 2 declared ‚Äî **regular (400)** for body copy, inputs, nav links, table/status text; **semibold (600)** for buttons/CTAs, the nav brand wordmark, active nav link, and page-level `Display`/`Heading` text. (Existing `btn` constant already uses 600 ‚Äî reused, not new.)

Usage map for Phase 3 elements:
- Nav brand ("monai") ‚Äî Heading, 20px/600, color `#e6e8eb`. Replaces the old page-level `<h1 style={{fontSize:28}}>` now that it lives in a persistent nav bar; keeps the wordmark readable at nav-bar scale.
- Nav links (Chat / Cashflow / Investments / Settings) ‚Äî Body size (14px), weight 400 inactive / 600 active, per Color contract below for the color swap.
- Page-level heading on `/settings` and `/investments` (e.g. "Settings", "Investments") ‚Äî Display, 28px/600.
- Settings card section titles ("LLM Provider & Model", "API Keys", "Preferences") ‚Äî Label style (12px/400, color `#9aa0a6`), matching the existing established pattern where every `card` section is headed by the `label` constant (e.g. "Log a transaction", "Recent transactions" in current `page.tsx`). Do not promote these to Heading size ‚Äî consistency with existing cards takes precedence.
- Form field labels, inline status/error text, masked-key hints ‚Äî Label, 12px/400.
- Input values, button labels, table cells ‚Äî Body, 14px.

## Layout & Component Notes (supplemental ‚Äî for planner/executor)

These are not part of the template's fixed sections but are load-bearing for implementation; included so the plan doesn't have to re-derive them.

**`<Nav/>` component** (`ui/app/components/Nav.tsx` or similar, rendered once in `ui/app/layout.tsx`):
- Client component (`"use client"`) ‚Äî uses `usePathname()` from `next/navigation` for active-link detection.
- Layout: horizontal flex row, brand wordmark left-aligned, four nav links right-aligned (or immediately following brand ‚Äî researcher default: brand left, links follow left-to-right in requirement order Chat/Cashflow/Investments/Settings, no right-alignment needed for a 4-link nav).
- Each link is a `next/link` `<Link>` (client-side transitions, satisfies UI-02's "no full page reload").
- Active state: text color Accent (`#3b82f6`) + weight 600 + 2px bottom border in Accent. Inactive: text muted (`#9aa0a6`) + weight 400, no border.
- Sticky top, `zIndex` above page content, bottom border `1px solid #2a2e37`, background Secondary (`#1a1d23`) to visually separate from the page's Dominant background.

**Page skeletons:**
- `/chat` ‚Äî existing ask box + SSE streaming + ProposalCard + tool trace section from current `page.tsx`, moved intact (no visual changes). Focal point: the ask box ‚Äî it sits first, is the only input control above the fold, and is where the eye should land on page load.
- `/cashflow` ‚Äî existing manual-entry form + recent-transactions table sections from current `page.tsx`, moved intact as interim content (full dashboard is Phase 4). Focal point: the recent-transactions card ‚Äî it carries the most data density and is the primary reason to visit this page today.
- `/investments` ‚Äî new skeleton: page Display heading "Investments are coming in Phase 5" inside a single `card`, with the empty-state body copy beneath it. No table, no form ‚Äî this is intentionally sparse per CONTEXT.md ("real skeleton page," not a stub route). Focal point: the skeleton headline card itself ‚Äî with no competing content on the page, it is the only thing to focus on.
- `/settings` ‚Äî new page: Display heading "Settings", then three `card` sections in order: "LLM Provider & Model" (provider dropdown + model text input + Save Provider button), "API Keys" (masked password-type inputs per configured provider + Save Keys button), "Preferences" (base currency input/select + price data source dropdown + Save Preferences button). Each section has its own inline success/error message per the Copywriting Contract. Focal point: the "LLM Provider & Model" card ‚Äî it is first in reading order and the setting most users come to this page to change.
- Root `/` ‚Äî redirect to `/chat` (server-side `redirect()` in `app/page.tsx`, per CONTEXT.md discretion note; a server-component redirect is simpler than a `next.config.js` rewrite and keeps routing logic colocated with the App Router).

**Settings form field types (from CONTEXT.md, locked):**
- Provider: `<select>` styled with the `input` constant ‚Äî options `ollama` / `claude` / `openai`.
- Model: `<input type="text">` ‚Äî pre-filled with the provider's current/default model on load and on provider change.
- API keys: `<input type="password">` ‚Äî placeholder shows masked value from server; empty value on submit means "keep existing."
- Base currency: `<input type="text">` (default `IDR`) ‚Äî free text is acceptable since currency validation is out of scope this phase (single-currency assumption holds per CLAUDE.md).
- Price data source: `<select>` ‚Äî options `coingecko` / `yfinance` / `manual`.
- All Save buttons: reuse `btn` constant; disabled + `"Saving‚Ä¶"` label while a request is pending (existing pattern from `addTx`/`ask` in `page.tsx`).
