## Spacing Scale

Standard 8-point scale only: **4, 8, 16, 24, 32, 48, 64** — unchanged from Phase 3, no new tokens introduced, no exceptions.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps, checkbox right-margin, label bottom-margin (reused) |
| sm | 8px | Button row gap, input-to-button gap, chart legend item gap, modal field-to-field vertical gap |
| md | 16px | Form grid gaps, dashboard widget-to-widget gap within a row, table cell horizontal padding |
| lg | 24px | Card padding (`card.padding`), card bottom-margin (`card.marginBottom`) — reused verbatim |
| xl | 32px | Page title bottom-margin, modal internal top/bottom padding |
| 2xl | 48px | Page outer padding (`main` horizontal/vertical, reused) |
| 3xl | 64px | Not used this phase |

Exceptions: none. New Phase 4 elements (dashboard grid, modals, chart containers) must use only the token scale above.

**Dashboard-specific:** the summary row (totals + per-account balances) and the two-chart row (donut + income/expense bar) use CSS grid with `gap: 16` (md token) between cards in the same row, and `marginBottom: 24` (lg token, i.e. reuse `card.marginBottom`) between rows. The trend chart spans the full width in its own row below.

**Chart container height (Pitfall 3, `04-RESEARCH.md`):** every `<ResponsiveContainer>` sits inside a `<div style={{ width: "100%", height: 280 }}>` with an explicit pixel height — 280px is a multiple of 8 (35 × 8) and matches the existing card's visual density. Do not rely on flexbox `flex: 1` for chart height.

**Modal-specific:** `TransactionModal` overlay uses a centered card, `maxWidth: 480`, `padding: 32` (xl token, more breathing room than a page card since it's the sole focus of a blocking overlay).

## Typography

Unchanged from Phase 3 — 4 declared sizes, 2 weights. New Phase 4 elements map onto the existing roles; no new sizes introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 400 | 1.4 |
| Body | 14px | 400 (600 for button/CTA text only) | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Weights: exactly 2 — **regular (400)** for body copy, inputs, table/status text, chart tick labels and tooltip content; **semibold (600)** for buttons/CTAs, dashboard figure values (see below), and the page-level `Display` heading.

Usage map for Phase 4 elements:
- Page heading "Cashflow" — Display, 28px/600 (matches `/settings` and `/investments` precedent from Phase 3).
- Dashboard summary figures (Total Income, Total Expenses, Net) — **Heading, 20px/600**, in the figure's semantic color (see Color below); the figure's caption ("Total Income") is Label style (12px/400, muted) directly above it. This is a deliberate exception to "Heading = page titles only" — these are the single most important glanceable numbers on the page and need to visually outrank body text without introducing a 5th size.
- Per-account balance rows (`current_balance` + `period_net`) — Body (14px/400) for the account name, Body/600 tabular-nums for both balance figures, consistent with the existing recent-transactions table's amount-column treatment (`fontVariantNumeric: "tabular-nums"`).
- Chart section titles ("Spending by Category", "Income vs Expense", "6-Month Trend") — Label style (12px/400, muted `#9aa0a6`), identical treatment to existing card headers ("Log a transaction", "Recent transactions") — do not promote to Heading size.
- Chart axis/tick labels and tooltip text — Label size (12px), Recharts default font-family inherited from the page (set via a `style` prop on `<Tooltip>`/`<XAxis>`/`<YAxis>` if Recharts doesn't inherit automatically — verify at implementation time).
- Modal field labels — Label, 12px/400 (reused `label` constant, no change).
- Category manager list rows (category name + affected-count badge) — Body, 14px/400 for name; affected-count badge is Label size (12px), muted, in a small pill (see Color).
- CSV upload result line ("Parsed 120 · Inserted 118 · Skipped 2") — Body, 14px/400, with the "Skipped" segment in the Destructive color if `skipped > 0`, else muted.

## Color

60/30/10 split — identical palette to Phase 3, no new hues invented. Recharts gets a small **derived** categorical palette (below) built from tints of the existing accent + supporting neutrals, so the donut/bar charts don't introduce off-brand colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f1115` | Page/body background, input field background, modal overlay backdrop (`rgba(15,17,21,0.72)`) |
| Secondary (30%) | `#1a1d23` | Card surfaces, modal card background, dashboard widget cards |
| Accent (10%) | `#3b82f6` | Reserved for: primary CTA buttons (Save/Add/Rename/Merge/Upload buttons), active period-selector pill, focus outline on inputs, the "income" bar/line series in charts |
| Destructive | `#f87171` | Reserved for: Delete buttons (transaction/account row + modal), destructive confirm-dialog primary action, negative/expense amounts, error inline text, the "expense" bar series in charts, "Skipped" count in CSV result when > 0 |

Accent reserved for: primary-action buttons only (Add transaction, Save account, Rename, Merge, Upload CSV), the active state of the period selector, input focus rings, and the "income" data series across all three charts. Accent is **not** used for card borders, section titles, chart gridlines, or decorative elements — matches the restraint established in Phase 3.

Supporting neutrals (reused, not new):

| Token | Value | Usage |
|-------|-------|-------|
| Border | `#2a2e37` | Card borders, table row dividers, modal card border, chart gridlines (Recharts `<CartesianGrid stroke="#2a2e37">`) |
| Text primary | `#e6e8eb` | Body text, input values, dashboard figure values before semantic coloring is applied |
| Text muted | `#9aa0a6` | Labels, section titles, chart axis ticks, placeholder text, category-manager affected-count badges |
| Success | `#4ade80` | Reserved for: positive net figures, "income" amounts in tables, save-success inline messages (existing pattern) — **also** the "income" bar series color (see chart palette note below) |
| Disabled | `#374151` | Disabled button background (Save/Add buttons while pending, reused) |

**Chart categorical palette** (for the spending-by-category donut, which needs N distinct slice colors beyond the 2 semantic income/expense colors): a fixed 6-color array derived from the existing palette family, declared once in `ui/app/styles.ts` as `chartColors`:

```
["#3b82f6", "#f87171", "#4ade80", "#fbbf24", "#a78bfa", "#22d3ee"]
```

(blue=Accent, red=Destructive, green=Success, plus 3 new supporting tints — amber, violet, cyan — used ONLY for category-donut slices beyond the first 3, never for buttons or UI chrome). If a user has more than 6 categories, cycle the array (`COLORS[i % COLORS.length]`) rather than adding more colors — this is the existing pattern already shown in `04-RESEARCH.md`'s Recharts code example.

**Income vs Expense bar chart:** income bars use Success (`#4ade80`), expense bars use Destructive (`#f87171`) — NOT Accent — because green/red income/expense coloring is the established convention already visible in the existing recent-transactions table (`t.amount < 0 ? "#f87171" : "#4ade80"`). Reusing that exact pair keeps the whole page internally consistent (table row colors match chart bar colors for the same semantic meaning).

**Trend chart (6-month):** two series — income (Success `#4ade80`) and expense (Destructive `#f87171`) as either grouped bars or two lines, consistent with the income/expense bar chart above. Net is not plotted as a third series (avoid a 3rd chart color) — net is understood from the totals row above the charts.

**New shared style constant — `dangerBtn`** (add to `ui/app/styles.ts`, spread over `btn`):
```ts
export const dangerBtn: React.CSSProperties = {
  ...btn,
  background: "#f87171",
};
```
Used for: Delete transaction / Delete account / Merge-confirm (destructive) buttons inside modals and the `ConfirmDialog`. Regular row-level "Edit"/"Delete" text-links in tables may instead use plain colored text (Destructive `#f87171` for Delete, muted `#9aa0a6` for Edit) rather than full buttons — see Layout notes.

## Layout & Component Notes (supplemental — for planner/executor)

These are not part of the template's fixed sections but are load-bearing for implementation; included so the plan doesn't have to re-derive them.

**Page structure (`ui/app/cashflow/page.tsx`, top to bottom):**
1. Page heading "Cashflow" (Display, 28px/600) + period selector (a row of pill buttons or a `<select>` styled with `input` — researcher default: pill buttons for `["This week", "This month", "Last month", "This year"]` since `resolve_period()`'s `PERIODS` tuple already names these; simpler than a dropdown for ≤4 options and matches the nav's existing flat, no-icon aesthetic).
2. **Summary row** — CSS grid, 3 columns on desktop (`gridTemplateColumns: "repeat(3, 1fr)"`, `gap: 16`): Total Income card, Total Expenses card, Net card. Each card: Label caption + Heading-styled figure value, figure colored Success/Destructive/text-primary respectively (Net uses Success if ≥0 else Destructive).
3. **Per-account balances row** — one `card` containing a compact table/list: account name, `current_balance` (Body/600, tabular-nums), `period_net` (Body/600, tabular-nums, Success/Destructive colored by sign) — D-04's dual-figure requirement.
4. **Charts row** — CSS grid, 2 columns on desktop (`gap: 16`): `CategoryDonut` card (left) + `IncomeExpenseBar` card (right). Each chart card header uses the existing Label-style section-title convention.
5. **Trend row** — full-width `card` containing `TrendChart`, ≥6-month rolling window (CASH-02), header "6-Month Trend".
6. **Transactions section** — existing recent-transactions table, extended with an "Add transaction" button (opens `TransactionModal` in create mode) in the section header row, and per-row "Edit" (opens modal in edit mode, pre-filled) / "Delete" (opens `ConfirmDialog`) text-links in a new trailing table column.
7. **Account manager section** — new `card`, header "Accounts", list of accounts with inline Edit/Delete actions + an "Add account" button; delete flow branches per D-06 (see Copywriting).
8. **Category manager section** — new `card`, header "Categories", list of category names (from `list_categories`) each with "Rename" and "Merge into…" actions; both fetch and display `affected_count` before the user confirms (D-09).
9. **CSV upload section** — new `card`, header "Import CSV", file input + "Upload CSV" button + result line (Copywriting above). Placement: last section on the page (researcher default per CONTEXT.md's "placement... open" — CSV import is an occasional/setup-time action, not a glanceable daily figure, so it belongs below the dashboard and CRUD sections, not competing with them for the fold).

**Focal point:** the summary row (Total Income / Total Expenses / Net) — it is first below the page heading, uses the largest colored figures on the page, and answers the single question "how am I doing financially" before any chart or table demands attention.

**`TransactionModal.tsx`** (D-10, shared create/edit):
- Centered overlay modal: backdrop `rgba(15,17,21,0.72)` covering the viewport, centered `card`-styled panel (`maxWidth: 480`, `padding: 32`).
- Fields (all reuse `input`/`label`): Date (`datetime-local`, reuse `toLocalDatetimeInputValue` pattern), Amount (`number`, reuse existing negative-is-expense convention + placeholder), Category (`text`), Merchant (`text`), Account (`text` — future-proof: could become a `<select>` of existing account names once `AccountManager` exists on the same page; researcher default is a `<select>` populated from the accounts list already loaded for the balances row, with a fallback "Add new…" free-text toggle only if needed — keep it simple as a `<select>` for this phase), Notes (`text`), Transfer (checkbox) — identical field set to the existing entry form.
- Footer: Cancel (plain muted text button) + primary action button (Accent `btn`, label "Add transaction" or "Save changes" per mode).
- Same component instance handles both modes; a `null` vs populated `editingTx` prop determines title ("Add transaction" vs "Edit transaction") and button label.

**`ConfirmDialog.tsx`** (D-03, Claude's discretion — shared across all destructive actions):
- Small centered modal, `maxWidth: 360`, `padding: 24` (lg token — smaller than the transaction modal since it holds only a question + 2 buttons).
- Structure: message text (Body, 14px/400), then a button row (Cancel = plain muted text button, Confirm = `dangerBtn`).
- Reused verbatim for: delete transaction, delete account (both branches), merge category. NOT used for rename category (non-destructive, no dialog per Copywriting above).

**`AccountManager.tsx`** (D-05/D-06):
- List rows: account name (Body), inline Edit (opens a small inline form or reuses a lightweight modal — researcher default: inline edit, turning the row into an `input` + Save/Cancel pair, since account edit is just a name change, not worth a full modal) and Delete (opens `ConfirmDialog`; if the delete attempt returns 422, replace the dialog's content with a destination-account `<select>` + the reassign-confirmation copy from the Copywriting Contract, then re-issue the delete with `reassign_to` set).

**`CategoryManager.tsx`** (D-09):
- List rows: category name (Body) + affected-count badge (small pill, Label size, muted background using Border color `#2a2e37` with `#9aa0a6` text, `borderRadius: 999`, `padding: "2px 8px"`) showing the live transaction count for that category.
- "Rename" opens an inline text input pre-filled with the current name; on submit, calls the affected-count endpoint first (already fetched for the badge, reused) then submits — no confirm dialog (non-destructive).
- "Merge into…" opens a `<select>` of the other category names; on selection, shows the `ConfirmDialog` with the merge copy (including affected count) before calling `POST /categories/merge`.

**`CsvUpload.tsx`** (CASH-08):
- Single `card`: `<input type="file" accept=".csv">` styled to match `input` visually (file inputs can't be fully restyled — wrap in a label styled as a button, or accept the native file-picker look with a small margin-top gap before it), "Upload CSV" button (disabled until a file is chosen), and the result line rendered after a successful `POST /import`.
- Loading state: button shows "Uploading…" (matches existing `"Saving…"` pattern) and is disabled during the request.

**Charts — shared conventions:**
- Every chart wrapped in `<div style={{ width: "100%", height: 280 }}>` before `<ResponsiveContainer>` (Pitfall 3).
- `<Tooltip>` formatter uses `Intl.NumberFormat` for IDR-style grouping (reuse the existing `fmt`/`Intl.NumberFormat` pattern already in `cashflow/page.tsx`, extended to omit `signDisplay: "always"` for chart tooltips where sign is already conveyed by color/label).
- `<CartesianGrid>` (bar/line charts) uses `stroke="#2a2e37"` (Border token), `strokeDasharray="3 3"`.
- Axis ticks and legend text use Label styling (12px, `#9aa0a6`) via `tick={{ fill: "#9aa0a6", fontSize: 12 }}`.
- No 3D, no gradients, no drop-shadows on chart elements — flat fills only, consistent with the existing flat/dark aesthetic (no skeuomorphism anywhere else in the app).

**Currency formatting (single-currency IDR, per CLAUDE.md constraint):** all monetary figures — dashboard totals, account balances, chart tooltips, transaction table amounts — format via `Intl.NumberFormat` with no explicit currency symbol prefix (existing convention shows plain signed numbers, e.g. `+50000` / `-25000`), consistent with the existing `fmt()` helper in `cashflow/page.tsx`. Do not introduce a `Rp` or `IDR` prefix this phase — that would be a scope-creep visual change beyond what Phase 3/existing code establishes; flag as a fast-follow if the user wants it, but it is not blocking for CASH-01..08.
