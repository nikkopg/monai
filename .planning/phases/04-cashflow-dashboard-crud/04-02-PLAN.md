  <behavior>
    - monthly_trend(months=6): returns {"tool": "monthly_trend", "rows": [...]} where each row is {month: "YYYY-MM", income, expense, net}; clamps months to a minimum of 6; window is CURRENT_DATE minus N months (rolling), so calling it in early calendar months still yields >=6 buckets that have data; excludes is_transfer rows.
    - account_balances(period_start, period_end): returns {"tool": "account_balances", "rows": [...]} where each row is {id, name, current_balance, period_net}; current_balance sums ALL that account's non-transfer transaction amounts (all-time, ignores the period); period_net sums only amounts inside [period_start, period_end); accounts with zero transactions appear with 0/0 (LEFT JOIN).
  </behavior>
  <action>
Add two module-level functions to `backend/tools.py`, following the existing aggregation shape verbatim (`with engine.connect() as c:`, parameterized `text()`, `float()` cast at the Python boundary — READ path, do NOT use Decimal here per Pitfall 6).

`monthly_trend(months: int = 6) -> dict`: clamp `months = max(months, 6)`. Use SQL `SELECT date_trunc('month', date) AS month, COALESCE(SUM(amount) FILTER (WHERE amount > 0), 0) AS income, COALESCE(SUM(-amount) FILTER (WHERE amount < 0), 0) AS expense FROM transactions WHERE is_transfer = false AND date >= date_trunc('month', CURRENT_DATE) - (:months || ' months')::interval GROUP BY 1 ORDER BY 1`. Bind `:months`. Build each row as `{"month": <first-7-chars of ISO month>, "income": float(r.income), "expense": float(r.expense), "net": float(r.income) - float(r.expense)}`. Return `{"tool": "monthly_trend", "rows": rows}`. The rolling `CURRENT_DATE - INTERVAL` window is load-bearing (Pitfall 4): do NOT use `date_trunc('year', ...)`.

`account_balances(period_start=None, period_end=None) -> dict`: reuse the existing `_date_clause` helper for the period predicate applied to `t.date`. SQL joins accounts to transactions: `SELECT a.id, a.name, COALESCE(SUM(t.amount), 0) AS current_balance, COALESCE(SUM(t.amount) FILTER (WHERE <period predicate on t.date>), 0) AS period_net FROM accounts a LEFT JOIN transactions t ON t.account_id = a.id AND t.is_transfer = false GROUP BY a.id, a.name ORDER BY a.name`. current_balance ignores the period (all-time, D-04); period_net applies the resolved period bounds. Build rows as `{"id": r.id, "name": r.name, "current_balance": float(r.current_balance), "period_net": float(r.period_net)}`. Return `{"tool": "account_balances", "rows": rows}`. Accept the already-resolved `(start, end)` tuple from `resolve_period()` — do not call `resolve_period` inside; the caller (Plan 03 endpoint) resolves once and passes the bounds.
