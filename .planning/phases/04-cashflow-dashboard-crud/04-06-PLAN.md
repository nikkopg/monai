<objective>
Close UAT gap 1 (major, test 3): the "This week" period pill breaks the entire /cashflow dashboard. The frontend sends `period=this_week`, but the backend named-period registry (`PERIODS`, backend/tools.py:30-33) has no week entry, so `resolve_period()` raises ValueError at tools.py:75 — and `GET /cashflow/summary` (backend/main.py:214-237) is the only resolve_period caller without the project-standard ValueError→HTTPException(422) mapping, so the user sees a raw 500 and the generic "Couldn't load the dashboard" error state. Diagnosed end-to-end in .planning/debug/this-week-period-fails.md.

Purpose: restores the UAT truth "Selecting the 'This week' period loads the dashboard with week-scoped figures" (04-UAT.md gap 1) and hardens the summary endpoint so invalid periods can never again surface as raw 500s.

Output: `this_week` and `last_week` named periods in backend/tools.py, a 422-mapped /cashflow/summary handler in backend/main.py, and regression tests in both backend test files.
</objective>
