## Spacing Scale

Standard 8-point scale only: **4, 8, 16, 24, 32, 48, 64** — unchanged from Phase 3/4, no new tokens introduced, no exceptions.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps, staleness-dot-to-timestamp gap, label bottom-margin (reused) |
| sm | 8px | Button row gap, input-to-button gap, holding-row action-link gap, staleness pill internal padding |
| md | 16px | Form grid gaps, platform-section-to-section gap, table cell horizontal padding |
| lg | 24px | Card padding (`card.padding`), card bottom-margin (`card.marginBottom`) — reused verbatim |
| xl | 32px | Page title bottom-margin, modal internal top/bottom padding (event-entry modal) |
| 2xl | 48px | Page outer padding (`main` horizontal/vertical, reused) |
| 3xl | 64px | Not used this phase |

Exceptions: none. New Phase 5 elements (platform groups, holding rows, event modal, override dialogs) must use only the token scale above.

**Platform-grouped holdings list layout:** each platform is its own `card` (not a flat table spanning all platforms) — this makes the per-platform subtotal a natural section footer and matches the "buckets" mental model established by `AccountManager`. Platform cards use `marginBottom: 24` (lg token, reuse `card.marginBottom`) between them. Inside a platform card, holding rows use a `<table>` with `borderCollapse: "collapse"`, row `borderTop: "1px solid #2a2e37"` (Border token) — the exact structural pattern already used by `AccountManager.tsx`'s account table.

**Portfolio total banner:** a single full-width `card` above the platform list, `padding: 24` (lg, reused), containing the total portfolio value (Display size) + "as of" timestamp (Label size, muted) + the "Refresh prices" button, laid out as a flex row with `justifyContent: "space-between"` and `alignItems: "center"`.

**Modal-specific (event-entry / override modals):** same shape as `TransactionModal` — centered overlay, `card`-styled panel, `maxWidth: 480`, `padding: 32` (xl token). `PriceOverrideDialog` (single price field + note) is smaller: `maxWidth: 360`, `padding: 24` (lg token), matching `ConfirmDialog`'s dimensions since it's a narrow single-purpose form, not a multi-field entry.

## Typography

Unchanged from Phase 3/4 — 4 declared sizes, 2 weights. New Phase 5 elements map onto the existing roles; no new sizes introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 400 | 1.4 |
| Body | 14px | 400 (600 for button/CTA text and P&L figures — see below) | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Weights: exactly 2 — **regular (400)** for body copy, inputs, table/status text, staleness timestamps; **semibold (600)** for buttons/CTAs, P&L/value figures, and the page-level `Display` heading.

Usage map for Phase 5 elements:
- Page heading "Investments" — Display, 28px/600 (matches the Phase 3 skeleton precedent and `/cashflow`'s "Cashflow" heading).
- **Total portfolio value** (banner) — Display, 28px/600, text-primary color (not semantic-colored — it's a magnitude, not a gain/loss). The "as of {time}" caption directly below is Label style (12px/400, muted).
- **Per-platform subtotal** (platform card header row) — Heading, 20px/600: platform name in text-primary, subtotal value in text-primary/600 tabular-nums, right-aligned in the same row. This is a deliberate exception to "Heading = page titles only," mirroring Phase 4's identical exception for dashboard summary figures — subtotals are the second-most-important glanceable number on the page.
- **Holding row fields** (ticker, quantity, avg cost, current price, current value) — Body, 14px/400 for ticker/asset-type/quantity/avg-cost/current-price; Body/600 tabular-nums for current value (it's a computed magnitude worth visual weight, same treatment as account balances in Phase 4).
- **P&L (IDR) and P&L%** — Body, 14px/600, tabular-nums, semantically colored (Success/Destructive — see Color). This is the one place two adjacent Body-weight figures sit side by side (absolute + percent), both bold, both colored identically per row so the eye reads them as one signal.
- **Staleness badge** ("Price as of {time}") — Label, 12px/400, muted `#9aa0a6` by default; when stale, the timestamp text itself does NOT change weight/size (avoid a 5th visual pattern) — only the color and leading dot glyph change (see Color).
- **Ticker/asset-type sub-line** (e.g. "BBCA · idx_stock") — Label, 12px/400, muted, directly under the ticker in the row (mirrors the existing pattern of a Label-style caption paired with a Body-style value elsewhere in the app).
- **Realized vs Unrealized P&L labels** (in the P&L summary sub-section) — Label, 12px/400, muted caption above each Heading-styled (20px/600) figure — identical caption+figure pairing as Phase 4's dashboard summary row (Total Income/Expenses/Net).
- **Modal field labels** — Label, 12px/400 (reused `label` constant, no change).
- **Event-ledger type selector** (buy/sell/dividend) — rendered as a `<select>` styled with `input`; option text is Body, 14px/400 (native select styling, not custom typography).
- **Platform manager list rows** — Body, 14px/400 for platform name, identical structure to `AccountManager`'s account rows.

## Color

60/30/10 split — identical palette to Phase 3/4, no new hues invented. No new chart palette needed (no new charts this phase).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f1115` | Page/body background, input field background, modal overlay backdrop (`rgba(15,17,21,0.72)`) |
| Secondary (30%) | `#1a1d23` | Card surfaces (portfolio-total banner, per-platform cards, modal cards) |
| Accent (10%) | `#3b82f6` | Reserved for: primary CTA buttons ("Log event", "Add platform", "Save platform", "Set price", modal submit buttons), "Refresh prices" button, input focus outline |
| Destructive | `#f87171` | Reserved for: Delete buttons (platform row + holding-override delete + event delete), destructive confirm-dialog primary action, negative P&L figures (loss), the stale-indicator dot when a price is past its TTL, error inline text |

Accent reserved for: primary-action buttons only (Log event, Add platform, Save platform, Set price, Refresh prices), input focus rings. Accent is explicitly **not** used for P&L coloring (gains use Success, not Accent — this keeps "blue = action you can take" and "green = good outcome" visually distinct, consistent with Phase 4's rule that Accent ≠ semantic income coloring), not used for card borders, section titles, or the staleness badge's fresh state (which is muted, not accent-colored, so it doesn't compete with real CTAs for attention).

Supporting neutrals (reused, not new):

| Token | Value | Usage |
|-------|-------|-------|
| Border | `#2a2e37` | Card borders, table row dividers, modal card border, staleness-pill background |
| Text primary | `#e6e8eb` | Body text, input values, portfolio total value, current-value figures (unsigned magnitudes) |
| Text muted | `#9aa0a6` | Labels, section titles ("Realized P&L", "Unrealized P&L" captions), ticker sub-line, fresh-state staleness timestamp, placeholder text |
| Success | `#4ade80` | Reserved for: positive P&L (IDR) and P&L% figures, gain-colored realized/unrealized summary figures — same green already used for positive net/income in Phase 4 |
| Disabled | `#374151` | Disabled button background (Refresh prices while in-flight, Save buttons while pending, reused) |

**Gains/losses color convention (accessibility requirement from design_scope — not color-only):** every P&L figure pairs color with an explicit sign character from `Intl.NumberFormat({ signDisplay: "always" })` (the exact `fmt()` pattern already used in `ui/app/cashflow/page.tsx`) — e.g. `+1,250,000` (Success, green) vs `-340,000` (Destructive, red). The `+`/`-` prefix is the non-color signal; color is reinforcement, never the sole differentiator. This satisfies WCAG 1.4.1 (use of color) using the exact mechanism Phase 4 already established for income/expense — no new pattern invented.

**Staleness indicator (INV-05, new this phase — not color-only):** a leading dot glyph (`●`) plus a text state, never a dot alone:
- **Fresh** (within TTL): muted dot `#9aa0a6` + "Price as of {time}" in muted Label text. No pill background — this is the default, low-attention state.
- **Stale** (past TTL): Destructive dot `#f87171` + "Price as of {time} · stale" in Destructive-colored Label text, inside a small pill (`background: "#2a2e37"` Border token, `borderRadius: 999`, `padding: "2px 8px"` — identical pill treatment to Phase 4's category-manager affected-count badge). The word "stale" is the non-color signal (readable by screen readers, visible without color perception); the dot+color is reinforcement only.

**Asset-type badge (informational, not semantic gain/loss):** a small muted pill (`background: "#2a2e37"`, `#9aa0a6` text, `borderRadius: 999`, `padding: "2px 8px"`, Label size) showing `crypto` / `idx_stock` / `mutual_fund` / `other` next to the ticker — reuses the exact category-manager badge visual, no new pill style invented.

## Layout & Component Notes (supplemental — for planner/executor)

These are not part of the template's fixed sections but are load-bearing for implementation; included so the plan doesn't have to re-derive them.

**Page structure (`ui/app/investments/page.tsx`, top to bottom):**
1. Page heading "Investments" (Display, 28px/600) — no period selector this phase (unlike `/cashflow`; portfolio value is a snapshot, not a period-scoped figure — D-05/D-06 explicitly defer the time-series view).
2. **Portfolio total banner** — one full-width `card`: "Total portfolio value" caption (Label) + Display-size figure + "as of {timestamp}" caption (Label, muted) on the left; "Refresh prices" button (Accent `btn`) on the right, flex row `justifyContent: "space-between"`.
3. **P&L summary row** — CSS grid, 2 columns on desktop (`gridTemplateColumns: "repeat(2, 1fr)"`, `gap: 16`, md token): "Unrealized P&L" card + "Realized P&L" card, each Label caption + Heading-styled (20px/600) signed figure, Success/Destructive colored by sign — identical structural pattern to Phase 4's 3-column summary row, just 2 columns since there's no "Net" equivalent this phase.
4. **Platform-grouped holdings** — one `card` per platform (not a single flat table), each containing:
   - Header row: platform name (Heading, 20px/600) + per-platform subtotal value (Heading, 20px/600, tabular-nums, right-aligned) + optional platform `kind` sub-caption (Label, muted, e.g. "brokerage").
   - A `<table>` of holding rows (mirrors `AccountManager`'s table structure): ticker + asset-type badge + staleness badge (first column, stacked), quantity, avg cost, current price, current value (Body/600 tabular-nums), P&L IDR (Body/600, signed, colored), P&L% (Body/600, signed, colored), trailing action-link column ("Log event" / "Set price" / "Edit" / "Delete" text-links, same visual treatment as `AccountManager`'s Edit/Delete links — muted for non-destructive, Destructive-red for Delete).
   - A holding whose net quantity is 0 is excluded from this render entirely (D-04 — "drops off the active list" is a display filter, backend still returns/retains the row).
   - Platform-with-zero-holdings still renders its card (empty-state body copy above) so the platform itself remains manageable (rename/delete) even before any holding is assigned to it.
5. **Unassigned holdings** (any holding with `platform_id IS NULL`) — rendered as one additional platform-shaped card titled "Unassigned" (Heading, 20px/600, muted color `#9aa0a6` instead of text-primary, to visually signal it's not a real platform) at the END of the platform list, same table structure. Only rendered if at least one such holding exists (researcher default — D-12 doesn't specify this case explicitly; a nullable `platform_id` per `05-RESEARCH.md`'s migration means this state is reachable and must not silently hide holdings).
6. **Platform manager section** — new `card`, header "Platforms", list of platforms with inline Edit/Delete actions + an "Add platform" button — structurally identical to `AccountManager.tsx` (D-12: "mirrors the Phase-4 account manager"); delete flow branches per the reassign-then-delete copy above.

**Focal point:** the portfolio total banner — it is first below the page heading, uses the largest figure on the page (Display size, same visual weight as Phase 4's page-level figures), and answers "what is my portfolio worth right now" before any per-holding detail demands attention. The Refresh-prices control sits in the same visual band so "is this number fresh" is answered in the same glance.

**`HoldingModal.tsx`** (event-entry form, D-01/D-04, primary INV-01 path):
- Centered overlay modal: backdrop `rgba(15,17,21,0.72)`, centered `card`-styled panel (`maxWidth: 480`, `padding: 32`) — identical shape to `TransactionModal`.
- Fields (all reuse `input`/`label`): Ticker (`text`), Asset type (`<select>`: crypto / idx_stock / mutual_fund / other), Platform (`<select>` populated from the platforms list, with a "(unassigned)" option), Event type (`<select>`: Buy / Sell / Dividend), Quantity (`number`, `step="any"`), Price (`number`, `step="any"`, IDR, label "Price per unit (IDR)"), Date (`datetime-local`, reuse `toLocalDatetimeInputValue` pattern verbatim from `TransactionModal`), Notes (`text`, optional).
- For `event_type === "dividend"`: relabel Quantity's neighboring Price field per the tool convention noted in `05-RESEARCH.md` ("quantity=1, price=amount for lump-sum dividends") — researcher default: keep both fields visible but change the Price label to "Dividend amount (IDR)" and default Quantity to `1` (still editable) when Dividend is selected, so the form doesn't grow a 5th conditional field.
- Footer: Cancel (plain muted text button) + primary action button (Accent `btn`, label "Log event").
- On success: closes and triggers a summary refetch (position recompute happens server-side per `05-RESEARCH.md` Pattern 2).

**`HoldingOverrideModal.tsx`** (D-03 direct escape hatch — visually and structurally distinct from `HoldingModal` so users don't confuse the two paths):
- Same modal shape (`maxWidth: 480`, `padding: 32`).
- Fields: Ticker, Asset type, Platform, Quantity, Avg cost (IDR), Purchase date, Currency (fixed "IDR", disabled/read-only input per D-07 — shown for transparency, not editable, since D-07 locks everything to IDR).
- A one-line warning caption (Label, muted, italicized via `fontStyle: "italic"`) above the form fields: "This directly sets the holding's position, bypassing the event ledger. Prefer \"Log event\" for a normal buy/sell." — makes the escape-hatch nature visible without a blocking dialog (researcher default — D-03 calls it an "exception path," so the UI should visually de-emphasize it, e.g. reachable via a smaller secondary text-link "Add holding directly" near the "Log event" primary button, not an equally-weighted second CTA).
- Footer: Cancel + primary action button (Accent `btn`, label "Add holding directly" / "Save holding" per create/edit mode) — same dual-mode-component pattern as `TransactionModal` (`editingHolding == null` → create, populated → edit).

**`PriceOverrideDialog.tsx`** (INV-04, manual price set):
- Small centered modal, `maxWidth: 360`, `padding: 24` (same dimensions as `ConfirmDialog`, since it's a narrow single-field form, not a multi-field entry).
- Fields: current price shown as read-only context text ("Current: {formatted price} · {source}"), then a single Price input (`number`, `step="any"`, label "New price (IDR)"), then the manual-price helper text from the Copywriting Contract above.
- Footer: Cancel (plain muted text) + primary action button (Accent `btn`, label "Set price").
- Reachable via the "Set price" text-link in each holding row's trailing action column.

**`PlatformManager.tsx`** (D-12, mirrors `AccountManager.tsx` exactly):
- Same component shape as `AccountManager`: inline-edit rows (row becomes an `input` + Save/Cancel pair on Edit click, matching `AccountManager`'s `editingId`/`editName` state pattern), Delete opens `ConfirmDialog`; if delete returns 422, swap the dialog's `children` slot for a destination-platform `<select>` + reassign copy, then re-issue with `?reassign_to=`.
- Additional field beyond `AccountManager`'s single `name`: an optional `kind` text input (e.g. "brokerage", "crypto_app", "fund_app") shown as a smaller secondary input next to Name in both the add-row and inline-edit-row layouts — Label-sized helper text "e.g. brokerage, crypto app" as the input's `placeholder`, not a separate caption (keeps the row height identical to `AccountManager`'s single-input row, per `05-RESEARCH.md`'s `platforms.kind` being `nullable=True`).

**`StalenessBadge.tsx`** (shared, INV-05 — new this phase, no existing precedent to reuse, built fresh per the Color section's spec above):
- Props: `fetchedAt: string | null`, `source: string`, `isStale: boolean` (server-computed per `05-RESEARCH.md`'s Architectural Responsibility Map: "TTL-vs-`fetched_at` comparison is business logic computed server-side and returned as a flag; browser only renders the badge/color" — the frontend never computes TTL math itself).
- Renders: `●` dot (colored per state) + "Price as of {relative time}" text, wrapped in the pill styling only when `isStale === true` (fresh state has no pill background, just the dot+text, to keep the default/common case visually quiet).
- Relative time formatting: researcher default — use a simple “X min ago” / “X hr ago” / absolute date-for-older-than-a-day helper (no new npm dependency; a ~15-line hand-rolled function is sufficient and matches the project's existing preference for minimal dependencies, per `05-RESEARCH.md`'s "Don't Hand-Roll" table not flagging this as a risk area).

**Refresh-prices interaction (D-09):**
- Button click → `POST /prices/refresh` (or equivalent) → button shows "Refreshing…" (disabled, `background: "#374151"` Disabled token) → on completion, refetch the investments summary so all rows/badges update in place, no page reload.
- Lazy per-page-load refresh (the other half of D-09) is invisible UI — it happens server-side on `GET /investments/summary` for any ticker whose cache is stale; the frontend doesn't need a loading state for this path beyond the page's normal initial-load skeleton, since it's folded into the same request that populates the page.

**Loading/initial state:** while `GET /investments/summary` is in flight on first mount, render a single centered `card` with muted Body text "Loading your portfolio…" (matches the minimal-skeleton precedent — no shimmer/spinner component exists elsewhere in the app, don't introduce one).

**Currency formatting (single-currency IDR, per CLAUDE.md constraint, D-07):** all monetary figures (portfolio total, per-holding avg cost/current price/current value, P&L IDR) format via the existing `Intl.NumberFormat` `fmt()` pattern from `ui/app/cashflow/page.tsx` — no `Rp`/`IDR` prefix (consistent with Phase 4's explicit deferral of that change). Quantity fields (especially crypto, `Numeric(28,8)` per `05-RESEARCH.md`) format with enough decimal precision to avoid visually rounding to zero — researcher default: up to 8 decimal places for `asset_type === "crypto"`, up to 2 for stocks/funds, trimming trailing zeros (e.g. `0.00341250` displays as `0.0034125`, not `0.00`).
