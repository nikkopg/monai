# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase does not introduce a new page or a new visual language — it extends the shipped `/investments` page (Phase 5, `05-UI-SPEC.md`) with (a) a currency field that becomes meaningful, (b) two new `asset_type` values (`cash`, `gold`) on existing forms, and (c) two new Recharts components (allocation pie, value/P&L history line) whose placement and interaction chrome were explicitly deferred here by Plans 03 and 04 ("placement, exact interactions, range presets, toggle rendering → `/gsd-ui-phase`"). Every token, color, spacing value, and copy convention below is inherited from `05-UI-SPEC.md` unless explicitly marked NEW. Nothing here reopens the shadcn/styling question — Phase 3/4/5 all declined it in the same milestone; re-asking a fourth time would violate "don't re-open locked decisions."

## Spacing Scale

Standard 8-point scale only: **4, 8, 16, 24, 32, 48, 64** — unchanged from Phase 3/4/5, no new tokens, no exceptions.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Toggle-pill internal vertical padding, range-button-to-button gap (tight variant), label bottom-margin (reused) |
| sm | 8px | Toggle-pill horizontal padding, chart-to-toggle-row gap, button row gap (reused) |
| md | 16px | Chart-section-to-section gap, form grid gaps (reused), gap between the two chart cards (pie + line) on desktop |
| lg | 24px | Card padding (`card.padding`), card bottom-margin (reused verbatim) |
| xl | 32px | Modal internal padding (reused, no change to existing modals) |
| 2xl | 48px | Page outer padding (reused) |
| 3xl | 64px | Not used this phase |

Exceptions: none.

**Chart placement (NEW — resolves Plans 03/04's deferred "exact placement"):** both charts render as their own full-width `card` sections, inserted into the existing page flow **between the P&L summary row and the "Log event" CTA row** (i.e. immediately after the existing Unrealized/Realized P&L grid, before the platform-grouped holdings list). Order top-to-bottom: **Allocation pie card, then Value/P&L history card** — allocation (a snapshot fact) precedes history (a trend), mirroring how the portfolio-total banner (a snapshot) already precedes the P&L summary before any per-holding detail. This placement means both charts are visible without scrolling past the holdings table, matching the "answer the big picture before the detail" principle `05-UI-SPEC.md` already established for the portfolio-total banner. Each chart card uses `marginBottom: 24` (lg token) between them, same as every other section on the page.

**AllocationPieChart card internal layout:** header row (`display: flex, justifyContent: space-between, alignItems: center, marginBottom: 16` — md token) containing the section title ("Allocation") on the left and the asset-type↔platform toggle (two-pill `<button>` group) on the right — same header-row shape as the existing platform-card header (title left, subtotal right). Below the header: the `ResponsiveContainer` wrapper (`width: 100%, height: 280` — load-bearing, per `04-RESEARCH.md`'s documented Recharts blank-render pitfall, verified again in this phase's own `07-PATTERNS.md`).

**ValueHistoryChart card internal layout:** header row identical shape — title ("Portfolio history") left, range-selector pill group (`1M` / `3M` / `6M` / `All`) right. Below the header: a second small toggle row (`marginTop: 8, marginBottom: 16` — sm/md tokens) for the value↔P&L view switch, left-aligned, Label-sized. Below that: the `ResponsiveContainer` wrapper (`width: 100%, height: 280`, same load-bearing pattern).

**Toggle/range-selector pill group (NEW component pattern, used by both charts):** a `<div style={{display: "flex", gap: 4}}>` (xs token) of `<button>` elements, each `padding: "4px 12px"`, `borderRadius: 6`, `fontSize: 12` (Label size), `border: "1px solid #2a2e37"` (Border token). Active pill: `background: "#3b82f6"` (Accent), `color: "white"`. Inactive pill: `background: "transparent"`, `color: "#9aa0a6"` (muted). This is a new visual pattern (no exact precedent in Phase 3/4/5) but is a direct composition of existing tokens only — no new color or spacing value invented.

## Typography

Unchanged from Phase 3/4/5 — 4 declared sizes, 2 weights (400 regular, 600 semibold). No new sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 400 | 1.4 |
| Body | 14px | 400 (600 for buttons/CTAs and value figures) | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Usage map for Phase 7 elements:
- **Chart card section titles** ("Allocation", "Portfolio history") — Heading, 20px/600, matching the platform-card header treatment (`05-UI-SPEC.md`'s exception for glanceable section titles).
- **Toggle/range-selector pill text** ("Asset type", "Platform", "1M", "3M", "6M", "All", "Value", "P&L") — Label, 12px/400 (600 has no additional pill state — active state is color-only + the existing sign convention doesn't apply here since these aren't gain/loss figures, so weight stays constant and only background/color differentiate active/inactive, consistent with `05-UI-SPEC.md`'s rule of avoiding a 5th visual pattern).
- **Pie slice tooltip values** — Body, 14px/400 inside the tooltip (Recharts default styling override, matching `CategoryDonut`'s existing `contentStyle`), IDR-formatted via the shared `fmtPlain`.
- **Line chart axis ticks** — Label, 12px/400, muted `#9aa0a6` (`tickStyle` constant, verbatim clone of `TrendChart.tsx`'s existing `tickStyle`).
- **Line chart tooltip values** — Body, 14px/400, IDR-formatted; P&L values additionally use the signed formatter (`fmtSigned`) matching every other P&L figure on the page.
- **Currency field** (`HoldingModal`/`HoldingOverrideModal`) — Label, 12px/400 for the field label ("Currency"); the `<select>` itself uses native Body 14px/400 option text, identical to the existing Asset type `<select>`.
- **Gold "grams" / "per-gram price" helper text** — Label, 12px/400, muted, same treatment as the existing CoinGecko-id helper caption in `HoldingOverrideModal`.
- **Cash entry helper text** — Label, 12px/400, muted, italic (matching the existing "This directly sets the holding's position…" warning caption pattern) — see Copywriting.

## Color

60/30/10 split — identical palette to Phase 3/4/5. **One new addition: the chart categorical palette is extended for asset-type grouping (5 groups now: crypto, idx_stock, mutual_fund, cash, gold, plus "other")** — reuses the exact `chartColors` array from `ui/app/styles.ts`, cycling `chartColors[i % chartColors.length]`, same as `CategoryDonut` already does for >6 spending categories. No new hex value invented.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f1115` | Page/body background, input field background, modal overlay backdrop (unchanged) |
| Secondary (30%) | `#1a1d23` | Card surfaces — now includes the two new chart cards |
| Accent (10%) | `#3b82f6` | Reserved for: all existing CTAs (unchanged) + **NEW: active toggle/range-selector pill background**, **NEW: portfolio-value line series color** (the primary line, matching Accent's "this is the main number" role — same reasoning `05-UI-SPEC.md` used for Display-size portfolio total) |
| Destructive | `#f87171` | Reserved for: all existing uses (unchanged) + **NEW: P&L line series when net-negative on the visible range is the dominant reading** — see P&L line color rule below |

Accent reserved for (extends the existing list): primary-action buttons, input focus rings, **active state of the asset-type↔platform toggle and the 1M/3M/6M/All range selector**, **the portfolio-value line in `ValueHistoryChart`** (a magnitude series, not a gain/loss series — Accent is correct here, distinct from the P&L line's semantic coloring below).

Supporting neutrals (reused, extended usage):

| Token | Value | Usage |
|-------|-------|-------|
| Border | `#2a2e37` | Card borders, table dividers (unchanged) + **NEW: inactive toggle-pill border, chart gridlines (`CartesianGrid stroke`, matching `TrendChart`'s existing value)** |
| Text primary | `#e6e8eb` | Body text, figures (unchanged) |
| Text muted | `#9aa0a6` | Labels, inactive toggle-pill text, chart axis ticks (unchanged pattern, new surface) |
| Success | `#4ade80` | Positive P&L (unchanged) + **NEW: P&L line series when the line is plotting the unrealized-P&L view and is currently ≥0** |
| Disabled | `#374151` | Disabled button background (unchanged) |

**P&L line color rule (NEW — resolves "like Bitget" two-view ambiguity):** the two `ValueHistoryChart` views use different, deliberate color semantics because they answer different questions:
- **Value view** (total portfolio value over time): single line, **Accent** `#3b82f6` — this is a magnitude, not a gain/loss signal, exactly matching `05-UI-SPEC.md`'s existing rule that "Accent is explicitly not used for P&L coloring" is preserved (the value line is not a P&L figure).
- **P&L view** (unrealized P&L over time): single line, colored by its **current/latest point's sign** — `Success` `#4ade80` if the most recent point in the visible range is ≥0, `Destructive` `#f87171` if negative. This mirrors every other P&L figure on the page (banner P&L cards, per-row P&L) using the identical sign→color rule, just applied to a line instead of a static number. Do NOT gradient/split the line's color mid-series (adds a visual pattern not used anywhere else in the app) — one flat color per render, re-evaluated when the range or the latest data point changes.

**Chart categorical palette for the asset-type pie (extends `chartColors`, no new hex):** `["#3b82f6", "#f87171", "#4ade80", "#fbbf24", "#a78bfa", "#22d3ee"]` cycling by index — deterministic order: whatever order the summary payload returns groups in (backend-determined, same as `CategoryDonut`'s existing behavior for category order). Same array is reused for the platform grouping when the toggle is set to "Platform."

## Layout & Component Notes (supplemental — for planner/executor)

These resolve the "exact placement / interactions / range presets / toggle rendering" items Plans 03 and 04 explicitly deferred to this pass.

**`AllocationPieChart.tsx`** (clone of `CategoryDonut.tsx`, per `07-PATTERNS.md`):
- Props: `data: { label: string; value: number }[]` (already-grouped current-IDR-market-value array from the summary payload — asset_type or platform grouping, selected by the parent's toggle state), no internal fetch.
- Structure: identical to `CategoryDonut` — `PieChart` > `Pie` (`dataKey="value"`, `nameKey="label"`, `innerRadius: 60`, `outerRadius: 100`, `paddingAngle: 2`) > `Cell` per slice cycling `chartColors`, `Tooltip` with the dark-theme `contentStyle` (verbatim clone: `background: "#1a1d23"`, `border: "1px solid #2a2e37"`, `borderRadius: 8`, `fontSize: 12`, `color: "#e6e8eb"`).
- Explicit-height wrapper (`div style={{width: "100%", height: 280}}` around `ResponsiveContainer`) is **load-bearing** — carried over from `04-RESEARCH.md`'s documented pitfall and reconfirmed in `07-PATTERNS.md`; omitting it renders a blank chart.
- Toggle state (`groupBy: 'asset_type' | 'platform'`) lives in the parent `page.tsx`, passed down as the `data` prop already resolved to the correct grouping — the component itself has no knowledge of "asset_type vs platform," it only renders whatever `{label, value}[]` array it receives (keeps the component a pure renderer, matching `CategoryDonut`'s existing statelessness).
- Empty state: if `data.length === 0`, render the muted "Add a holding to see your allocation." Body text instead of `ResponsiveContainer` (matches the page's existing "No holdings yet" tone, avoids rendering an empty/broken pie).

**`ValueHistoryChart.tsx`** (clone of `TrendChart.tsx`, `Bar`→`Line`, per `07-PATTERNS.md`):
- Props: `data: { date: string; total_market_value: number; total_pnl: number }[]` (the `GET /investments/history` response, already range-filtered server-side per the selected range preset — the component does not re-filter client-side; changing the range re-fetches).
- Structure: `LineChart` (not `BarChart`) with `CartesianGrid stroke="#2a2e37" strokeDasharray="3 3"` (unchanged from `TrendChart`), `XAxis dataKey="date"` with `tickStyle`, `YAxis` with `tickStyle`, `Tooltip` (same dark `contentStyle`), and **one** `Line` element whose `dataKey` and `stroke` switch based on the view toggle: `dataKey="total_market_value" stroke="#3b82f6"` for the Value view, `dataKey="total_pnl" stroke={latestPnl >= 0 ? "#4ade80" : "#f87171"}` for the P&L view (per the P&L line color rule above). Only one `Line` renders at a time — the two views are mutually exclusive tabs, not overlaid series (keeps the chart legible on a narrow page, matches "like Bitget" which shows one metric at a time with a switcher, not two overlapping y-axes).
- Range selector: four `<button>` pills (1M/3M/6M/All) in the toggle-pill pattern defined above; clicking a pill re-fetches `GET /investments/history?range={token}` (range tokens map to the backend's accepted set from Plan 04 — the component doesn't invent its own range math, it passes the token through).
- Explicit-height wrapper — same load-bearing pattern, `width: 100%, height: 280`.
- Empty/sparse-history state: if `data.length < 2`, render the muted "Not enough history yet — check back after a few days of price snapshots." Body text instead of the chart (a single point can't draw a meaningful line; this also correctly handles the D-13 no-backfill collector-just-went-live case without a misleading flat/broken render).

**`page.tsx` wiring:**
- Fetch `GET /api/investments/history?range=3M` alongside the existing `platforms`/`investments/summary` `Promise.all` on mount (extend the existing `load()` callback, one more parallel fetch — do not serialize it after the summary fetch).
- Re-fetch history only when the range selector changes (not on every summary refresh) — a separate `useCallback` (`loadHistory(range)`) distinct from the existing `load()`, to avoid re-fetching history on every "Refresh prices" click (which only touches `price_cache`, not `portfolio_value_history`).
- The asset-type↔platform toggle for the pie needs no fetch — both groupings are already present on the one `investments/summary` payload (per `07-03-PLAN.md`'s explicit "no new backend endpoint" constraint) — the toggle only switches which array of the already-fetched payload is passed to `AllocationPieChart`.

**`HoldingModal.tsx` and `HoldingOverrideModal.tsx` — currency + cash/gold changes:**
- `ASSET_TYPES` constant extends from `["crypto", "idx_stock", "mutual_fund", "other"]` to `["crypto", "idx_stock", "mutual_fund", "cash", "gold", "other"]` in both files (currently only enumerated in `HoldingOverrideModal.tsx`; `HoldingModal.tsx` should be checked for the same array during implementation and kept in sync).
- Currency field: changes from `<input value="IDR" readOnly disabled />` to an editable `<input>` (free-text, uppercased on blur, 3-4 char, matching the backend's `^[A-Z]{3,4}$` validation — client-side validation is a UX nicety, the backend is the source of truth per correctness-by-construction) OR a `<select>` of common currencies (IDR, USD, USDT, EUR, SGD — researcher default: a short curated `<select>` with an "Other…" option that reveals a free-text input, avoiding a bare free-text field that invites typos on every entry — same escape-hatch UX pattern already used for the CoinGecko-id field). Recommendation: **`<select>` with curated list + "Other…" free-text fallback** — cheaper to get right than validating arbitrary free text inline, and covers the realistic case (the user's own crypto-in-USD + IDR + USDT per `07-CONTEXT.md`'s "Specific Ideas") without over-building a full ISO-4217 picker.
- When `assetType === "cash"`: the "Quantity" field label becomes "Balance" (per Copywriting), the "Avg cost" field is **hidden** (cash has no cost basis per CG-01 — showing a disabled/greyed field would be confusing; omit it entirely, following the existing conditional-field precedent the dividend-amount relabeling already established in `HoldingModal`), and the cash helper text (muted italic, above the fields, same position/style as the existing D-03 warning caption) explains the no-ledger semantics.
- When `assetType === "gold"`: the "Quantity" field label becomes "Grams", and a muted helper caption appears near the Price/Set-price affordance explaining the manual-per-gram convention (Copywriting above) — no field is hidden for gold, since it uses the full ledger like any other asset type (CG-02).
- These conditional-field patterns (label swap, field hide/show, contextual helper text) directly extend the existing `event_type === "dividend"` conditional-relabeling pattern already shipped in `HoldingModal.tsx` per `05-UI-SPEC.md` — no new interaction paradigm, same `if (assetType === X)` conditional-render approach.

**`PriceOverrideDialog.tsx` — gold label extension:**
- When overriding a gold holding's price, the field label reads "New price per gram (IDR)" instead of "New price (IDR)" — a one-line conditional on `holding.asset_type === "gold"`, everything else in the dialog (dimensions, footer, helper text position) unchanged from `05-UI-SPEC.md`'s existing spec.

**Loading state for charts:** while `GET /investments/history` is in flight, the `ValueHistoryChart` card shows the same minimal muted "Loading…" pattern already used for the page's initial summary load (no shimmer/spinner — consistent with the project's stated preference for minimal loading UI). The pie chart has no separate loading state since it derives from the already-loaded summary payload (no independent fetch).
