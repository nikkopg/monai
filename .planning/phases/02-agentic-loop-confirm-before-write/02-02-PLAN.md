<objective>
Add the confirm-before-write vertical slice: write tools in `tools.py` that PRODUCE `proposals`
rows (never mutate target data), a confirm endpoint that atomically applies a proposal's batch
payload + writes audit_log rows + invalidates the agent cache, plus reject and pending-list
endpoints. Wire the new write tools into the agent's tool list so a chat write request flows
end-to-end: agent → proposal row → user confirm → atomic write + audit. The agent stream also
surfaces the confirm token to the originating chat session so the inline ProposalCard (Plan 03)
can approve the write.

Purpose: Delivers CHAT-04 (show change, write nothing until approved), CHAT-05 (single-use
operation-scoped token), CHAT-06 (audit log), CHAT-07 (add/edit/delete transactions, accounts,
categories, holdings via confirm flow), and D-06 (orphan-delete blocked + explained).
Output: `get_session_sync`, the `propose_*` tool family, Proposal schemas, three new proposal
endpoints, agent wiring that surfaces `proposal_id` + `proposal_token` in the answer event, and
full lifecycle tests (create/confirm/reject/expire/replay/audit/orphan-block).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| chat → write tool | Untrusted intent crosses into a state-changing proposal-producer |
| client → confirm endpoint | A single-use token + API key guard the only path that mutates data |
| proposal → target tables | Mutation happens server-side, atomically, with audit, only on valid confirm |
| agent_stream → originating client | The confirm token is delivered ONLY to the proposing chat session via the SSE answer event |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Repudiation | confirm endpoint | mitigate | Status==pending checked BEFORE token compare; second confirm → 409; test_token_single_use (CHAT-05, Pitfall 3) |
| T-02-05 | Tampering | confirm endpoint | mitigate | 15-min server-side expiry (`expires_at`, tz-aware); confirm after expiry → 410; test_expired_proposal (D-09/D-10) |
| T-02-06 | Elevation of Privilege | proposal token | mitigate | `secrets.token_urlsafe(32)` (256-bit) + unique DB index + `hmac.compare_digest` constant-time; wrong token → 401 |
| T-02-07 | Information Disclosure | ProposalOut / GET /proposals | mitigate | `token` field excluded from ProposalOut; never serialized in GET; test_get_proposals_excludes_token asserts it. Token surfaces ONLY in the agent_stream answer event to the originating client (single-use, 15-min TTL) and is kept out of the visible tool-call trace |
| T-02-08 | Spoofing | confirm/reject endpoints | mitigate | `dependencies=[Depends(require_api_key)]` on both; test_confirm_requires_api_key asserts 401 without key |
| T-02-09 | Tampering | propose_delete_account/category | mitigate | Orphaning deletes blocked at tool level — no proposal row created; test_orphan_delete_blocked (D-06) |
| T-02-10 | Tampering | write tools (tools.py) | mitigate | Tools are proposal-producers only; grep gate asserts no target-table mutation inside propose_* (Pitfall 5) |
| T-02-11 | Repudiation | confirm executor | mitigate | One AuditLog row per affected row, before/after JSONB, in the same commit (CHAT-06) |
